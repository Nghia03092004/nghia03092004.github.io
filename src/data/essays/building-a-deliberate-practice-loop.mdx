The biggest trap in technical practice is confusing **time spent** with **skill acquired**.

When I solve problems inconsistently, I get short bursts of intensity and then long stretches of drift. The fix is not motivation. The fix is a loop with enough structure to survive low-energy days.

## The loop

I keep the cycle small:

1. Pick one problem or one implementation detail.
2. Write down the constraint that makes it difficult.
3. Solve it.
4. Record the idea that made the solution unlock.
5. Revisit the note later and compress it further.

That last step matters. A notebook full of raw logs becomes noise surprisingly fast. Compression is where the lesson hardens.

## What gets recorded

I only keep three things after a session:

- the wrong assumptions
- the decisive insight
- the smallest reusable pattern

Anything beyond that has to justify its existence.

> A good archive is not a dump. It is a set of decisions about what deserves to remain visible.

## Why this site is structured in streams

Essays, code notes, and Project Euler logs age differently.

- Essays keep context and arguments.
- Code notes keep implementation details.
- Euler notes keep the shape of a solve.

Trying to mix them in one generic timeline usually makes all three worse. Separate streams make retrieval easier without needing a heavy publishing system.

## A practical default

If I do not know what to publish after a working session, I start with this template:

```txt
Problem:
Constraint:
Wrong turn:
Unlock:
Reusable pattern:
```

That is usually enough to turn one afternoon of work into a note worth keeping.
