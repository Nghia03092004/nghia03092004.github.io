Not every snippet should become a permanent article.

I sort technical notes by how quickly they decay:

## Fast decay

These are notes tied to a one-off migration, a specific version mismatch, or a throwaway debugging path.

They can live in a scratch file or a dated log.

## Medium decay

These are implementation patterns that might stay useful for a few months.

I keep them as code notes with:

- the exact context
- the decision taken
- the reason the obvious alternatives were rejected

## Slow decay

These become essays or canonical write-ups because the underlying pattern persists.

Examples:

- how I scope a small tool
- how I structure a practice loop
- how I review solution notes after solving a problem

The classification is rough, but it prevents the archive from becoming a junk drawer.
