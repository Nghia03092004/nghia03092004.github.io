# Project Euler Sources

This document explains the source model for the Project Euler section of the site.

## Current canonical location

The website now treats `project_euler/problem_XXX/` inside this repository as the canonical per-problem bundle used for maintenance and deployment.

Each problem directory may contain:

- `solution.md`
- `solution.tex`
- `solution.cpp`
- `solution.py`
- `statement.html`

## Where the current material came from

The website repository already contained the active rendered write-ups and code:

- `solution.md`
- `solution.cpp`
- `solution.py`
- `statement.html` when mirrored statements had already been synced

During the Project Euler subsystem reorganization, real editable LaTeX sources were restored from the external archive:

`C:\Users\ADMIN\Downloads\combine\project_euler_unified`

That archive included a `website_manifest.json` file plus canonical `solution.tex` files for the modern per-problem structure. Those real LaTeX files were copied into this repository without rewriting the mathematical content.

## Source-of-truth policy

- `statement.html` represents the mirrored official Project Euler statement prepared for web rendering.
- `solution.tex` represents the editable mathematical source when available.
- `solution.md` remains the active website rendering layer used by Astro today.
- `solution.cpp` and `solution.py` remain the code artifacts shown on each problem page.

## Why both `solution.tex` and `solution.md` exist

The current Astro site renders from `solution.md`, not directly from LaTeX.

That is an implementation detail of the current website, not a claim that Markdown is the only meaningful source. Restoring `solution.tex` makes the archive easier to maintain long term, because the rigorous mathematical write-up can exist as an editable source file alongside the rendered website layer.

For now, the intended relationship is:

- maintain the web page through `solution.md`
- preserve `solution.tex` as the editable mathematical source when it exists
- keep the two conceptually aligned
- do not rewrite one into a simplified form of the other

## Statement workflow

`statement.html` is generated by the Project Euler statement sync tooling under `tools/project-euler/`.

That workflow mirrors the official problem statement and copies required media into:

- `project_euler/problem_XXX/statement.html`
- `public/project-euler-media/`

`statement.html` should normally be regenerated by tooling rather than hand-edited.

## Future editing guidance

- Edit `solution.cpp` and `solution.py` directly when code changes are needed.
- Edit `solution.tex` directly when maintaining the mathematical source and a real LaTeX file exists.
- Update `solution.md` carefully so the website rendering remains aligned with the intended mathematical content.
- Do not use placeholder LaTeX files for problems that do not yet have a real source file.
- Keep Problem 761 especially stable; it is already a benchmark entry for the archive’s rigorous style.

## Supporting files

- `project_euler/STRUCTURE.md` documents editorial structure rules.
- `project_euler/TAXONOMY.md` documents the controlled Project Euler tags.
- `project_euler/metadata.json` stores per-problem taxonomy tags.
