\section*{Overview}

Heavy-light decomposition is the standard reduction from tree path queries to array interval queries. It breaks the
tree into a small number of heavy chains, then uses a segment tree or Fenwick tree on the flattened order.

When a problem keeps asking for values on paths between arbitrary vertices, HLD is often the first general technique
that turns the tree into something interval-based and manageable.

\section*{When to Use It}

Use it when:

\begin{itemize}
  \item queries are on paths between arbitrary vertices,
  \item path answers can be merged associatively,
  \item the tree is static,
  \item you can maintain each chain with a range-query structure.
\end{itemize}

It is usually unnecessary for pure subtree queries, where an Euler-tour flattening alone often solves the problem.

\section*{Core Idea}

For each vertex, choose one child with maximum subtree size as its \textbf{heavy child}. Heavy edges link a vertex to
its heavy child; the other edges are light.

\includegraphics[width=\textwidth]{figures/chains.svg}

Following heavy edges downward forms vertex-disjoint chains. Every vertex belongs to exactly one chain, and each chain
becomes a contiguous interval in the flattened array.

\section*{Key Insight}

The crucial fact is not the chain picture itself. The crucial fact is:
\textbf{every time a root-to-node path crosses a light edge, the remaining subtree size at least halves}.

That means a root-to-node path crosses only \(O(\log n)\) light edges. Therefore a path between two arbitrary vertices
breaks into only \(O(\log n)\) heavy-chain segments.

That is the entire reason HLD is useful: one hard tree path becomes a logarithmic number of easy interval queries.

\section*{Operations / Main Technique}

The usual workflow is:

\begin{itemize}
  \item first DFS: compute subtree sizes and choose the heavy child,
  \item second DFS: assign each vertex a chain head and a position in the flattened array,
  \item build a segment tree over the flattened vertex values,
  \item answer a path query by repeatedly climbing the deeper chain head.
\end{itemize}

\paragraph{Worked path query.}
While \(\texttt{head[u]} \neq \texttt{head[v]}\), query the deeper chain segment, jump that endpoint to the parent of
its chain head, and continue. Once both vertices are on the same chain, finish with one final interval query.

\section*{Correctness Intuition}

Every tree edge is either heavy or light, so any path is partitioned into consecutive chain segments with no overlap
and no gaps. Each of those chain segments is contiguous in the flattened array by construction.

Therefore the path answer is exactly the merge of those segment-tree answers. The logarithmic number of segments comes
from the ``light edge halves the subtree'' argument.

\section*{Complexity Analysis}

\begin{itemize}
  \item preprocessing: \(O(n)\),
  \item point update: \(O(\log n)\),
  \item path query with a segment tree: \(O(\log^2 n)\),
  \item subtree query in the same flattening: \(O(\log n)\),
  \item memory: \(O(n)\) plus the inner range structure.
\end{itemize}

The extra logarithm in path queries comes from doing \(O(\log n)\) chain jumps and paying \(O(\log n)\) per segment
tree query.

\section*{Implementation}

The reference code stores values on vertices and supports:

\begin{itemize}
  \item build from initial vertex values,
  \item point assignment,
  \item path-sum query,
  \item subtree-sum query.
\end{itemize}

That is the version I want to trust first. For edge-value problems, the common trick is to store each edge's value on
its deeper endpoint and adjust the final same-chain interval accordingly.

\section*{Common Pitfalls}

\begin{itemize}
  \item Forgetting whether values live on vertices or on edges.
  \item Comparing depths of vertices instead of depths of chain heads in the climb loop.
  \item Getting the final same-chain interval off by one.
  \item Reusing subtree intervals without checking that the flattening order is compatible.
  \item Writing a correct segment tree but assigning \texttt{head} or \texttt{pos} in the wrong DFS order.
\end{itemize}

\section*{Variants / Extensions}

\begin{itemize}
  \item edge-value HLD,
  \item path updates via lazy segment trees,
  \item path max, min, xor, or custom associative states,
  \item mixing path queries with subtree queries in the same flattened order,
  \item HLD plus LCA or binary lifting for related tree tasks.
\end{itemize}

Compared with link-cut trees, HLD wins on simplicity when the represented tree is static.

\section*{Practice Problems}

\begin{itemize}
  \item Path sum or path maximum queries with point updates.
  \item Tree problems that mix subtree sums and arbitrary path queries.
  \item Static-tree tasks where dynamic-tree machinery would be unnecessary.
\end{itemize}
