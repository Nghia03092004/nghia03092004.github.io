\section*{Overview}

The convex hull trick is the DP optimization pattern for transitions that can be rewritten as evaluating a family of
lines at query points:
\[
dp[i] = \min_j (m_j x_i + b_j)
\]
or the same with \(\max\).

The big reduction is that each old state becomes a line, and each new state becomes a query. Once that translation is
visible, many quadratic DP transitions collapse to logarithmic or even amortized constant time per state.

\section*{When to Use It}

Use it when:

\begin{itemize}
  \item the transition can be rearranged into \(m x + b\),
  \item many candidate lines accumulate over time,
  \item query points or slopes are monotone, or you have another data structure for dynamic lines,
  \item the naive DP is too slow.
\end{itemize}

If the problem does not reduce cleanly to lines and point queries, trying to force CHT usually creates more confusion
than speed.

\section*{Core Idea}

Each transition source contributes one line \(y = m x + b\). For a target state with query value \(x\), the answer is
the lowest line value at that \(x\).

\includegraphics[width=\textwidth]{figures/lower-hull.svg}

So the optimization problem becomes a geometry/data-structure problem: maintain the lower envelope of useful lines and
query it quickly.

\section*{Key Insight}

There are really two families of CHT:

\begin{itemize}
  \item \textbf{monotone hulls}: slopes are inserted in sorted order and queries come in sorted \(x\)-order,
  \item \textbf{dynamic hulls}: insertion/query order is arbitrary, so a stronger structure such as Li Chao tree is
  needed.
\end{itemize}

The deque-based hull in this note belongs to the first family. It is extremely fast, but only because the monotonicity
assumptions let every popped line disappear forever.

\section*{Operations / Main Technique}

For the monotone minimum version:

\begin{itemize}
  \item \textbf{add line}: insert lines in monotone slope order,
  \item \textbf{remove bad line}: if the middle of three lines is never optimal, pop it from the back,
  \item \textbf{query}: while the second line beats the first at the current \(x\), pop from the front.
\end{itemize}

\paragraph{Worked DP pattern.}
If
\[
dp[i] = a_i^2 + \min_{j < i}(dp[j] + b_j^2 - 2 a_i b_j),
\]
then each \(j\) contributes slope \(m_j = -2b_j\) and intercept \(c_j = dp[j] + b_j^2\), queried at \(x = a_i\).

\section*{Correctness Intuition}

Every line that survives on the lower hull owns an interval of \(x\)-values where it is optimal. Those intervals
appear in the same order as the lines on the hull.

If inserting a new line makes the intersection order stop being monotone, the middle line loses its whole ownership
interval and can never be optimal again. Likewise, if the second line is already better than the first at the current
query \(x\), the first line will never return for any later monotone query.

That is why both kinds of popping are permanent.

\section*{Complexity Analysis}

For the monotone deque version:

\begin{itemize}
  \item add line: amortized \(O(1)\),
  \item query: amortized \(O(1)\),
  \item total for \(n\) insertions and \(n\) monotone queries: \(O(n)\),
  \item memory: \(O(n)\).
\end{itemize}

General online variants usually cost \(O(\log n)\) per operation instead.

\section*{Implementation}

The reference code stores a deque of lines and uses cross multiplication with \texttt{\_\_int128} rather than floating
intersections. That choice is deliberate: integer geometry is much easier to trust under contest constraints.

The exposed interface is minimal:

\begin{itemize}
  \item \texttt{add\_line(m, b)},
  \item \texttt{query(x)}.
\end{itemize}

\section*{Common Pitfalls}

\begin{itemize}
  \item Using the monotone deque version when slopes or query points are not monotone.
  \item Mixing up minimum-hull and maximum-hull inequalities.
  \item Comparing line intersections with floating point when exact integer comparison is possible.
  \item Ignoring equal-slope edge cases.
  \item Deriving the data structure correctly but deriving the DP transformation incorrectly.
\end{itemize}

\section*{Variants / Extensions}

\begin{itemize}
  \item Li Chao tree for arbitrary insertion/query order,
  \item offline hulls after sorting queries,
  \item maximum instead of minimum,
  \item hulls inside divide-and-conquer or alien-trick style optimizations.
\end{itemize}

Li Chao tree is the extension I keep in mind when the geometry is still linear but the monotonicity assumptions are
gone.

\section*{Practice Problems}

\begin{itemize}
  \item DP recurrences with one quadratic term that expands into lines.
  \item Problems where sorted coordinates give monotone query points for free.
  \item Tasks that can be solved either by monotone CHT or by Li Chao tree.
\end{itemize}
